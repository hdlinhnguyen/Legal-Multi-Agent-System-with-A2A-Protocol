💡 Architecture Comparison

Capability Stage 4 (In-Process Graph) Stage 5 (Distributed A2A Network)
Deployment Topology Single Python Process / Shared execution memory. Multiple microservices running on unique ports/hosts.
Discovery & Registry Hardcoded in-memory mappings or imports. Dynamic Service Discovery using a Registry on startup.
Distributed Tracing Single-process execution flow (easier local debugging). `trace_id` and `context_id` header propagation across HTTP calls.
Fault Isolation One failure kills the whole process. Surviving nodes complete the query even if Tax Agent dies.

⚙️ Kiến trúc Stage 4: Single-Process LangGraph StateGraph

Trong Stage 4, toàn bộ pipeline chạy trong một Python process duy nhất. Các agent là các nodes trong LangGraph StateGraph, giao tiếp qua shared LawState dictionary — không có HTTP, không có serialization.

✅ Ưu điểm
  • Không có network overhead (~0ms inter-agent)
  • Dễ debug — single call stack
  • Shared memory state, no serialization
  • Dễ test với pytest
❌ Hạn chế
  • Single point of failure — 1 agent crash = toàn bộ sập
  • Không thể scale agent riêng lẻ
  • Không thể dùng khác ngôn ngữ/runtime
  • Khó deploy lên cloud microservices

🌐 Kiến trúc Stage 5: Distributed A2A Microservices

Stage 5 triển khai mỗi agent như một FastAPI microservice độc lập giao tiếp qua Google A2A Protocol (JSON-RPC 2.0 over HTTP). Agents tìm nhau qua Service Registry.

Registry:10000
Customer Agent:10100
Law Agent:10101
Tax Agent:10102
Compliance Agent:10103
🔑 A2A Protocol — Key Concepts
  • Agent Card: manifest tại /.well-known/agent.json
  • context_id: nhận dạng conversation session
  • trace_id: distributed tracing toàn pipeline
  • Task lifecycle: submitted → working → completed
⚡ Optimizations áp dụng
  • Persistent httpx.AsyncClient (connection pool)
  • Agent card cache (không re-fetch manifest)
  • Registry discover cache TTL=60s
  • LLM singleton qua @lru_cache

⚡ Fault Tolerance — Khi Tax Agent sập

Stage 5 có khả năng tự phục hồi khi một specialist agent lỗi. Law Agent catch exception từ call_tax và điền fallback message vào state thay vì crash toàn bộ pipeline.
[FAULT] [law_agent] call_tax: ConnectError: [Errno 61] Connection refused :10102
[FAULT] [law_agent] call_tax failed → tax_result = "[Tax analysis unavailable]"
[FAULT] [law_agent] Compliance result OK. Continuing aggregate()...
[FAULT] [law_agent] ✓ Partial response returned (compliance only)
Stage 4 vs Stage 5 khi có lỗi
Tax Agent lỗi Stage 4: ❌ Exception propagate lên main
Stage 5: ✅ Caught, fallback, continues
